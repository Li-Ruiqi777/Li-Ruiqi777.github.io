核心类

TensorRT 中有多个关键类，分别用于模型解析、构建、优化和推理执行

ILogger

ILogger 是 TensorRT 中日志记录的接口，用于输出 TensorRT 的运行时日志，帮助开发者调试和跟踪模型的加载、优化和执行过程。

  • ILogger类是个抽象类，实际使用时必须创建个子类并重写其log()方法
  • 创建IRuntime、IBuilder类都需要一个ILogger示例，因为需要记录日志

IRuntime

IRuntime 是 TensorRT 中用于加载和管理推理引擎的类。它负责从序列化的 TensorRT 引擎文件中恢复出引擎对象。

  • 主要方法：
    • createInferRuntime：创建一个推理运行时实例
    • deserializeCudaEngine：从字节流中反序列化出推理引擎

ICudaEngine

ICudaEngine 是 TensorRT 中的推理引擎类，表示一个可以执行推理任务的模型引擎。它由 IBuilder 构建并由 IRuntime 反序列化。

  • 主要方法：
    • createExecutionContext：为推理任务创建执行上下文
    • getBindingIndex：根据名称(string)获得某个输入/输出在所有binding中的索引
    • getBindingDimensions：获取某个binding的维度，如果是动态维度，则返回的Dim对象对应位是-1

binding：指模型的一个输入或输出节点，类似ONNX的ValueInfoProto类

IExecutionContext

IExecutionContext 是 TensorRT 中用于执行推理的上下文类。它负责管理推理执行的输入输出数据，并提供执行推理的接口。

  • 主要方法：

    • execute：执行推理任务

    • executeV2：执行带有多个输入输出的推理任务

    • get/setBindingDimensions

可以看到推理引擎和和上下文都可以获得binding的维度，但是上下文还可以设置binding的维度（主要用于动态维度的情况）。如果一个模型支持==动态维度==，则在推理上下文中必须先给binding设置一个具体的维度

cudaStream_t

cudaStream_t代表一个流对象。在CUDA中，每个流表示一系列在 GPU 上按顺序执行的操作。在同一个流中的操作会按照提交的串行执行（不需要显示调用同步函数），而不同流中的操作又并行执行，前提是没有依赖关系

流中的串行计算任务中的每个指令都被GPU拆成了多个线程并行运行在多个核心上，这是GPU的SIMT（Single Instruction Multy Data）模型。当 GPU 中的线程数大于核心数时，GPU 会通过 调度 来有效管理这些线程的执行，与 CPU 类似，GPU 会进行线程挂起和切换。但是GPU的并行行体现在多个核心同时执行任务，而不是一个核心并发处理多个任务

流的类型

  1. 默认流：所有不指定流的 CUDA 操作都会被提交到默认流（cudaStreamDefault）。默认流中的操作会串行执行，意味着它们会按照提交的顺序依次执行。
  2. 用户定义的流：通过 cudaStreamCreate() 创建的流，它们允许在多个流中并行执行操作。这些流的操作不需要严格的顺序约束，可以在不依赖的情况下并行处理，从而提高 GPU 的利用率。

常见操作

  • 创建流：cudaStreamCreate 用于创建一个流
  • 销毁流：cudaStreamDestroy 用于销毁流
  • 同步流：
    • 通过 cudaStreamSynchronize 方法等待某个流中的所有操作完成
    • 通过cudaDeviceSynchronize方法等待所有流中的所有操作完成

显存管理

1.在设备显存中分配一块指定大小的内存

  • devPtr：设备内存的指针，将返回设备显存的地址（指针变量不仅可以保存内存地址，也可以保存显存的地址）
  • size：要分配的内存大小（单位：字节）

数据在显卡和内存间的拷贝

1.同步数据拷贝，它会阻塞 CPU 执行直到数据拷贝完成

  • dst：目标内存地址（可以是主机内存或设备内存）
  • src：源内存地址（可以是主机内存或设备内存）
  • count：要拷贝的字节数
  • kind：指定拷贝的方向（cudaMemcpyHostToDevice, cudaMemcpyDeviceToHost, cudaMemcpyDeviceToDevice, cudaMemcpyHostToHost）

2.用于异步数据拷贝，不会阻塞 CPU 执行，通常配合 CUDA 流使用

  • stream：指定异步操作所使用的流

一个模型显存占用太大怎么解决

首先要弄清楚显存的占用由哪几部分：

  • 模型权重：这部分不是大头，因为即使模型只有几KB、实际消耗显存也可能由几GB
  • Excution Context：这是显存消耗的主要部分，每次创建推理上下文的时候，TensorRT会预计这个模型计算时所需要的总显存，然后分配

为什么权重那么小，却需要这么多显存？

模型包含很多算子，所以有很多中间计算结果，这些中间结果都会占用显存的

  • 动态分配的显存：用于在显卡上接收输入数据，以及在显卡上保存结果

总上，一个模型显存占用太大的主要原因是推理上下文，那么解决这个问题有以下途径：

  • 不使用动态维度
  • 使用量化技术（–fp16或–int8）
  • 减小缓冲区大小（–workspace）
  • 每次推理时动态创建和销毁推理上下文，而不是将其一直保存在内存和显存中
